## 1. Arquitectura de Software

### 🔴 CRÍTICO

#### 1.1 Páginas monolíticas de 450–525 líneas con lógica de negocio embebida

**Evidencia:** `Hoy.tsx` (470 loc), `MiSemana.tsx` (523 loc), `Objetivos.tsx` (525 loc), `Planificacion.tsx` (452 loc), `Tablero.tsx` (419 loc).

Cada página contiene simultáneamente: definición de estado local (6–12 `useState`), lógica de mutación con `useMutation`, handlers async, lógica de permisos (`puedeEditarTarea`, `puedeGestionarTarea`) y ~200 líneas de JSX con estilos inline de Tailwind.

**Impacto:** Violación grave de SRP y SoC. Cualquier cambio de UI, lógica de negocio o flujo de permisos requiere modificar el mismo archivo, aumentando el riesgo de regresión.

**Remediación:**
- Extraer lógica de mutaciones y queries a custom hooks por página (`useHoyPage`, `useMiSemanaPage`).
- Extraer secciones de JSX a componentes presentacionales (`TareasPlanificadasSection`, `IncidenciasSection`, `BitacoraSection`).
- Centralizar lógica de permisos en un hook `usePermisos(tarea)`.

---

#### 1.2 Ausencia total de tests (0 archivos de test)

**Evidencia:** Búsqueda de `*.test.*` y `*.spec.*` en `src/` devuelve 0 resultados. No hay `vitest`, `jest`, `@testing-library` ni `playwright` en `package.json`.

**Impacto:** No existe red de seguridad para refactors ni validación de reglas de negocio críticas (marcado de atrasadas, permisos jefe/miembro, transiciones de estado).

**Remediación:**
- Agregar `vitest` + `@testing-library/react`.
- Priorizar tests unitarios para: `lib/tableroEstado.ts`, `lib/tareaEstado.ts`, `lib/semanas.ts`, `lib/fecha.ts`.
- Tests de integración para hooks: `useTareas`, `useTablero`, `useMiSemana`.
- Tests e2e con Playwright para flujos: login → crear tarea → completar.

---

### 🟡 IMPORTANTE

#### 1.3 Función `parseTarea` duplicada en 4 archivos

**Evidencia:** Definida idénticamente como `function parseTarea(row): Tarea { return row as unknown as Tarea; }` en `api/semana.ts`, `api/tablero.ts`, `api/hoyColumnas.ts`, `hooks/useTareas.ts`. Similar para `parseEvento` (3 archivos) y `parseNota`.

**Remediación:** Centralizar en `lib/parsers.ts` o `types/parsers.ts`.

---

#### 1.4 Mapas de labels/badges de estado duplicados en ≥5 archivos

**Evidencia:** `estadoLabel`, `estadoBadge`, `estadoPillClass`, `overlayBadgeClass`, `estadoTareaBadge` se definen parcial o totalmente en `TaskItem.tsx`, `Tablero.tsx`, `Planificacion.tsx`, `Objetivos.tsx`, `DraggableTareaTablero.tsx`.

**Remediación:** Crear `lib/estadoConfig.ts` con una sola fuente de verdad: `{ label, badgeClass, pillClass, color }` por cada `EstadoTarea`.

---

#### 1.5 CSS inline en bloque `<style>` dentro de `Login.tsx`

**Evidencia:** `Login.tsx` líneas 80–195 contienen un bloque `<style>{...}</style>` de 115 líneas con CSS en bruto (animaciones, layout, campos). Es el único componente con este patrón.

**Impacto:** Inconsistencia en la estrategia de estilos, sin scope ni deduplicación.

**Remediación:** Mover a `styles/login.css` importado como módulo, o convertir a clases Tailwind/MC.

---

#### 1.6 Clase CSS `mc-badge-info` usada pero nunca definida

**Evidencia:** Usada en 6 archivos (`Tablero.tsx`, `Objetivos.tsx`, `Bitacora.tsx`, `ModalDetalleTareaSemana.tsx`, `DraggableTareaSemana.tsx`, `DraggableTareaTablero.tsx`) pero no existe definición en `index.css` ni `tokens.css`.

**Impacto:** Bug visual — los badges "En progreso" no tienen estilos de color.

**Remediación:** Agregar a `index.css`:
```css
.mc-badge-info {
  background: color-mix(in srgb, var(--mc-color-accent) 12%, transparent);
  color: var(--mc-color-accent);
}
```

---

#### 1.7 Mezcla de Tailwind utilities y CSS custom sin convención clara

**Evidencia:** Los componentes usan clases `mc-*`, Tailwind utilities (`flex`, `gap-4`), y CSS variables inline (`style={{ fontSize: 'var(--mc-text-lg)' }}`) sin documentación de cuándo usar cada enfoque.

**Remediación:** Documentar convención y eliminar `style={}` en favor de clases.

- **1.10** Sin capa de abstracción sobre InsForge database (Repository pattern mejoraría testabilidad).

## 2. UX / Product Design

### 🔴 CRÍTICO

#### 2.1 Sin soporte responsive real en vistas principales

**Evidencia:** Vista semanal usa `md:grid-cols-6` con `min-w-[960px]` en Kanban. Sidebar flotante de 52px sin toggle en mobile.

**Impacto:** App inutilizable en móviles. Técnicos TI necesitan consultar agenda desde teléfono.

**Remediación:** Sidebar → bottom nav en mobile. Semana → lista vertical con swipe. Kanban → tabs por columna.

---

### 🟡 IMPORTANTE

- **2.2** Modales sin trap de foco ni cierre por Escape (falla WCAG 2.1 AA §2.1.2). Crear componente `<Modal>` reutilizable.
- **2.3** Feedback de carga inconsistente. Sin skeleton loaders ni spinners consistentes.
- **2.4** Menús contextuales (⋯) pueden cortarse fuera del viewport. Usar `@floating-ui/react`.
- **2.5** Sin confirmación visual de drag & drop exitoso. Agregar highlight en zona destino.

### 💡 SUGERENCIA

- **2.6** Botón "+" poco descriptivo — usar texto "Nueva incidencia" con ícono.
- **2.7** Emojis de carga (🟢🟡🔴) no accesibles para screen readers.
- **2.8** Sin modo oscuro — tokens existen pero solo para tema claro.

## 3. Web Performance

### 🟡 IMPORTANTE

#### 3.1 Google Fonts bloqueante en el render path

**Evidencia:** `index.html` carga Inter con `<link>` síncrono, bloqueando First Contentful Paint.

**Remediación:** Agregar `rel="preload"` + `as="style"`.

---

#### 3.2 Sin code splitting — todo el bundle es un solo chunk

**Evidencia:** `vite.config.ts` sin `manualChunks` ni `React.lazy()` en rutas. Las 10 páginas se cargan en el bundle inicial.

**Remediación:** `const Planificacion = lazy(() => import('@/pages/Planificacion'));`

---

#### 3.3 Invalidación excesiva de queries

**Evidencia:** Cada mutación invalida 3–5 query keys manualmente, disparando refetches innecesarios.

**Remediación:** Usar `refetchType: 'active'` o query keys jerárquicas.

---

### 💡 SUGERENCIA

- **3.4** Verificar cache HTTP headers en Vercel.
- **3.5** `tailwind.config.js` tiene `content: []` — verificar tree-shaking de clases.
- **3.6** Imágenes de logo sin dimensiones explícitas (riesgo CLS).

---

## 4. Seguridad (OWASP)

### 🔴 CRÍTICO

#### 4.1 Operaciones multi-paso sin transaccionalidad

**Evidencia:** `eliminarTareaConMotivo` inserta en `log_accion` y luego `DELETE` en `tarea`. Si el DELETE falla, queda log huérfano. Similar en `bloquearTareaConLog`, `reprogramarTareaConLog`, `desbloquearTareaConLog`.

**Remediación:** Usar funciones RPC de InsForge con transacciones PostgreSQL.

---

#### 4.2 RLS policies para miembros no verificadas/ausentes

**Evidencia:** Solo `003_rls_policies_jefe.sql` da acceso total al jefe. No hay archivo visible de RLS para miembros.

**Impacto:** Un miembro podría acceder/modificar datos de otros usuarios via PostgREST.

**Remediación:** Crear/verificar políticas: `USING (asignado_a = auth.uid())` en `tarea`, `USING (usuario_id = auth.uid())` en `evento`.

---

### 🟡 IMPORTANTE

- **4.3** Validación de inputs solo client-side (justificación mín. 10 chars). Agregar CHECK constraints en BD.
- **4.4** `asegurarUsuario` crea usuario automáticamente sin whitelist de email/dominio.
- **4.5** Sin headers de seguridad HTTP en `vercel.json` (CSP, X-Frame-Options, etc.).

### 💡 SUGERENCIA

- **4.6** Fetch interceptor — considerar rate limiting client-side.
- **4.7** `console.error` con datos sensibles en producción (`api/semana.ts:301`).

## 5. Code Review / Calidad

### 🔴 CRÍTICO

#### 5.1 Type safety inexistente en parsers — `as unknown as Tarea`

**Evidencia:** Todos los parsers usan double-cast que anula TypeScript completamente.

**Impacto:** Cambios de schema de BD no detectados en compilación; errores solo en runtime.

**Remediación:** Usar Zod schemas para validación runtime.

---

### 🟡 IMPORTANTE

- **5.2** Funciones async sin captura de error (`catch { toast.error('...') }` sin loggear). Sin Error Boundaries.
- **5.3** Hooks exportan funciones standalone (no-hooks) — mover a `api/` o `services/`.
- **5.4** Comentarios residuales `// ← agregar` en producción (`TaskItem.tsx`).
- **5.5** Complejidad ciclomática alta: `Hoy.tsx` tiene 12 useState + 8 funciones async.

### 💡 SUGERENCIA

- **5.6** Sin Prettier ni formateo consistente.
- **5.7** Mezcla de CRLF y LF en archivos.
- **5.8** Solo existe `Button.tsx` como componente atómico de UI — crear `Input`, `Select`, `Badge`, `Modal`.

### 🔴 CRÍTICO

#### 6.1 Sin meta tags, Open Graph, ni favicon

**Evidencia:** `index.html` solo tiene `<title>Nexora</title>`. Sin description, og:*, icon, theme-color.

---

- **6.3** Elementos `<div role="button">` sin `aria-pressed`/`aria-expanded` donde aplique.

- **6.5** Secciones sin `<section aria-label>`.
- **6.6** Contraste `--mc-color-text-secondary` sobre `--mc-color-bg` ratio ~4.6:1, ajustado para textos <12px.
