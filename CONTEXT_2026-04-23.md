## 2. Arquitectura General

```
Monorepo simple
├── db/               # Schema SQL, migraciones, seed
│   ├── schema.sql    # DDL consolidado (7 tablas + RLS + función)
│   ├── seed.sql      # Datos de prueba
│   └── migrations/   # Incrementales (002–004)
└── web/              # SPA React (Vite)
    └── src/
        ├── api/          # Capa de acceso a datos (InsForge SDK)
        ├── components/   # UI organizada por dominio
        ├── hooks/        # Lógica de negocio por página/feature
        ├── lib/          # Utilidades, config, parsers, cliente InsForge
        ├── pages/        # Páginas lazy-loaded
        ├── providers/    # AuthProvider, AppProviders (QueryClient, Router, Toaster)
        ├── store/        # Zustand (authStore)
        ├── styles/       # CSS adicional (sprint4, tokens)
        └── types/        # Tipos TypeScript del dominio
```

### Flujo de arranque

1. `main.tsx` → `setAppIcons()` → instala fetch interceptor → monta `<StrictMode>`
2. `AppErrorBoundary` (class component) envuelve todo
3. `AppProviders`: valida env vars → `QueryClientProvider` → `BrowserRouter` → `AuthProvider` → `<App />`
4. `AuthProvider`: llama `insforge.auth.getCurrentUser()` → `asegurarUsuario()` (upsert en `public.usuario`) → actualiza `authStore`
5. `App`: define `<Routes>` con rutas públicas y protegidas

### Patrón de capas

```
Page (componente visual)
  └→ useXxxPage (hook de orquestación: usa React Query + lógica UI)
       └→ api/xxx.ts (acceso a InsForge SDK, parsers Zod)
            └→ InsForge PostgREST / RPC
```

## 3. Módulos Funcionales

### 3.1 Hoy (`/hoy` — `Hoy.tsx`)
- Vista diaria del usuario autenticado.
- Columnas: Tareas del día, Incidencias (imprevistos), Eventos, Notas de bitácora rápidas.
- Hook: `useHoyPage` + `useHoyColumnas`.
- API: `hoyColumnas.ts` — queries por fecha y usuario.

### 3.2 Mi Semana (`/semana` — `MiSemana.tsx`)
- Grilla L–V + sábado/domingo con tareas planificadas del usuario.
- Backlog de tareas libres en panel lateral.
- **Drag & drop** (@dnd-kit): mover tareas entre días y desde backlog.
- Crear tareas planificadas y libres, eventos.
- Hook: `useMiSemanaPage` + `useMiSemana`.
- API: `semana.ts`.
- Componentes: `DraggableTareaSemana`, `SemanaDiaDrop`, `ModalDetalleTareaSemana`, `ModalMiSemana`.

### 3.3 Planificación (`/planificacion` — `Planificacion.tsx`) 🔒 Solo jefe
- Vista multi-miembro: carga semanal de todos los miembros activos.
- Asignación de tareas a miembros específicos por día.
- Notas de semana (tabla `configuracion_semana`).
- Hook: `usePlanificacionPage`.
- API: `planificacion.ts`.

### 3.4 Tablero Kanban (`/tablero` — `Tablero.tsx`)
- 4 columnas: Pendiente, En progreso, Bloqueada, Completada.
- Filtros: por usuario, por objetivo, mostrar/ocultar completadas.
- **Drag & drop** entre columnas con justificación obligatoria para bloquear/cancelar.
- Lógica de "estado efectivo": tarea planificada vencida → se muestra como atrasada.
- Hook: `useTableroPage` + `useTablero`.
- API: `tablero.ts`.
- Componentes: `ColumnaKanban`, `DraggableTareaTablero`.

### 3.5 Objetivos (`/objetivos` — `Objetivos.tsx`)
- CRUD de objetivos con responsable asignable.
- Vista de tareas vinculadas por objetivo.
- Progreso calculado (% completadas / total no canceladas).
- Hook: `useObjetivosPage`.
- API: `objetivos.ts`, `objetivosMetricas.ts`.

### 3.6 Bitácora (`/bitacora` — `Bitacora.tsx`)
- Notas de texto libre con visibilidad configurable.
- **Conversión**: nota → tarea planificada, nota → evento.
- Jefe ve notas de todo el equipo; miembro solo las propias.
- Hook: `useBitacora`.
- API: `bitacora.ts`.
- Componentes: `ModalConvertirTarea`, `ModalConvertirEvento`.

### 3.7 Métricas (`/metricas` — `Metricas.tsx`)
- KPIs por usuario: activas, completadas 7d, objetivos activos, atrasadas.
- KPIs por rango: desglose por estado y por semana ISO.
- Comparativa entre miembros.
- Hook: `useMetricas` + `useObjetivosMetricas`.
- API: `objetivosMetricas.ts`.

### 3.8 Órdenes de Trabajo (`/ordenes-trabajo` — `OrdenesTrabajo.tsx`)
- Flujo formal: Borrador → Pendiente → Aprobada → En ejecución → Completada (o Rechazada/Cancelada).
- Número correlativo generado por RPC (`generar_numero_ot`).
- Tipos de trabajo configurables (CRUD por jefe).
- Datos de cierre: receptor (nombre, DNI, cargo).
- Impresión de OT con formato (`OTImpresion.tsx`).
- Hook: `useOrdenesTrabajoPage`.
- API: `ordenTrabajo.ts`.
- Componentes: `OTFormModal`, `OTImpresion`.

### 3.9 Autenticación
- Login con email/password (InsForge Auth).
- Flujo de recuperación: Forgot → Verify Code → Reset Password.
- Whitelist de dominios de email (configurable via env).
- Auto-provisioning: primer login crea fila en `public.usuario` con rol `miembro`.
- Fetch interceptor: reintento automático en 401 + refresh de sesión.

## 4. RBAC — Control de Acceso por Rol

### Roles

| Rol | Código BD | Descripción |
|-----|-----------|-------------|
| **Jefe** | `'jefe'` | Supervisor del equipo. Acceso total. |
| **Miembro** | `'miembro'` | Técnico del equipo. Acceso limitado a sus datos. |

### Matriz de permisos

| Recurso / Acción | Jefe | Miembro |
|-------------------|------|---------|
| Ver todas las tareas | ✅ | Solo propias |
| Crear tareas para otros | ✅ | Solo auto-asignación |
| Planificación multi-miembro | ✅ | ❌ (ruta protegida) |
| Aprobar/rechazar OT | ✅ | ❌ |
| Ver bitácora del equipo | ✅ (todas) | Solo propias |
| Ver notas `solo_jefe` | ✅ | ❌ |
| Crear objetivos | ✅ (asigna a cualquiera) | ✅ (solo auto-responsable) |
| Métricas comparativas | ✅ | ✅ (limitadas) |
| Gestionar tipos de trabajo OT | ✅ | ❌ |

### Implementación

**Frontend:**
- `ProtectedRoute`: redirige a `/login` si no hay sesión.
- `JefeRoute`: redirige a `/hoy` si `rol !== 'jefe'`.
- `selectEsJefe(authStore)`: selector para condicionales en UI.

**Backend (RLS en PostgreSQL):**
- Función `sgtd_es_jefe()`: verifica rol del `auth.uid()` en tabla `usuario`.
- Políticas `sgtd_jefe_*_all`: acceso total (`FOR ALL`) a las 7 tablas para usuarios con rol jefe.
- Miembros: sin políticas explícitas de lectura propia definidas en el schema consolidado (pendiente de implementar políticas de miembro).

## 5. Esquema de Base de Datos

### Diagrama ER

```
usuario (PK: id uuid = auth.uid())
  ├── 1:N → objetivo (creado_por, responsable_id)
  ├── 1:N → tarea (asignado_a, creado_por)
  ├── 1:N → evento (usuario_id)
  ├── 1:N → nota_bitacora (usuario_id)
  └── 1:N → log_accion (usuario_id)

objetivo
  └── 1:N → tarea (objetivo_id, ON DELETE SET NULL)
```

## 6. Flujos Principales

### 6.1 Ciclo de vida de una Tarea

```
                  ┌──────────────┐
                  │   CREACIÓN   │
                  └──────┬───────┘
                         ▼
              ┌── planificada ──┐
              │  no_planificada │
              │     libre       │
              └────────┬────────┘
                       ▼
                  [pendiente]
                   ▼      ▼
          [en_progreso] [bloqueada]* ──→ [desbloquear]
                   ▼                          │
             [completada]                     ▼
                                        [pendiente]
                   
  * Requiere justificación ≥10 caracteres
  
  En cualquier momento:
    → [reprogramada]* (nueva fecha + justificación)
    → [cancelada]* (justificación obligatoria)
    → [atrasada] (calculado: planificada && fecha < hoy && estado abierto)
```

### 6.2 Ciclo de vida de una Orden de Trabajo

```
[borrador] ──enviar──→ [pendiente] ──aprobar(jefe)──→ [aprobada]
                           │                              │
                      rechazar(jefe)              iniciar ejecución
                           │                              │
                           ▼                              ▼
                      [rechazada]                   [en_ejecucion]
                                                         │
                                                   completar(receptor)
                                                         │
                                                         ▼
                                                    [completada]

  En cualquier momento (pre-completada): → [cancelada]
```

### 6.3 Flujo de Autenticación

```
1. Usuario abre app → AuthProvider.bootstrap()
2. getCurrentUser() → ¿sesión válida?
   ├─ NO → clear() → ProtectedRoute redirige a /login
   └─ SÍ → asegurarUsuario(authUser)
            ├─ ¿Existe en public.usuario? → retorna existente
            └─ ¿No existe?
                 ├─ ¿Email en dominio permitido? → INSERT con rol='miembro'
                 └─ ¿Dominio no permitido? → throw Error
3. setAuth(authUser, usuario) → app carga
```

### 6.4 Conversión Nota → Tarea/Evento

```
1. Usuario crea nota en bitácora
2. Selecciona "Convertir en tarea" o "Convertir en evento"
3. Completa formulario (título, fecha, prioridad/tipo)
4. Sistema crea la entidad destino
5. Marca nota.convertida_en = 'tarea' | 'evento'
```

## 7. Decisiones de Diseño

### Arquitectura
- **SPA sin SSR**: app interna corporativa, no requiere SEO. Deploy en Vercel como static.
- **InsForge como BaaS**: equivalente a Supabase. Auth + PostgREST + RPC. Sin backend custom.
- **Monorepo mínimo**: `db/` y `web/` sin tooling de monorepo (no Turborepo/Nx).
- **Fetch interceptor**: parche global de `window.fetch` para auto-refresh de tokens JWT en 401.

### Estado y Datos
- **Zustand solo para auth**: estado de sesión global. Todo el server state via React Query.
- **React Query por módulo**: cada hook de página crea sus queries con keys estructuradas.
- **Parsers Zod con fallback**: `safeParse` retorna raw cast en caso de error para no romper UI.

### UI/UX
- **Design system "Meta Canvas"**: tokens CSS custom (`--mc-color-*`, `--mc-font`), badges, pills.
- **Lazy loading de páginas**: cada ruta usa `React.lazy()` + `Suspense`.
- **Error boundaries por sección**: `SectionErrorBoundary` envuelve cada ruta; `AppErrorBoundary` global.
- **Drag & drop nativo**: @dnd-kit para semana y tablero Kanban.
- **Justificación obligatoria**: bloquear, cancelar, reprogramar y eliminar tareas requieren ≥10 caracteres.

### Seguridad
- **RLS habilitado en todas las tablas** (7 tablas).
- **Jefe = acceso total via RLS**: función `sgtd_es_jefe()` como SECURITY DEFINER.
- **Whitelist de dominios**: configurable via env, validada en client antes del INSERT.
- **Security headers en Vercel**: CSP, X-Frame-Options DENY, nosniff, strict referrer, permissions policy.
- **Robots noindex/nofollow**: app interna.

### Convenciones
- **Semana ISO `YYYYWW`**: formato texto, calculado con `semanaIsoDesdeFecha()`.
- **Fechas locales `YYYY-MM-DD`**: para `fecha_planificada` (date en BD).
- **Fechas UTC ISO**: para timestamps (`created_at`, `fecha_completada`).
- **Idioma español en dominio**: nombres de tablas, columnas, tipos, labels.
- **Path alias `@/`**: mapea a `web/src/` via tsconfig + vite.
