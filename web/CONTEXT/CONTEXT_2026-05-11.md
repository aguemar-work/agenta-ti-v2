## 1. Stack tecnológico

| Capa | Tecnología |
|------|------------|
| UI | React 19 + TypeScript (~6), Vite 8 |
| Routing | React Router 7 |
| Datos remotos | TanStack Query 5 + `@insforge/sdk` (Postgres, RLS) |
| Estado local | Zustand 5 |
| Validación runtime | Zod 4 (`src/lib/schemas.ts`) |
| Estilos | Tailwind **3.4** + tokens CSS (`DESIGN-SYSTEM` / `mc-*`) |
| Toasts | Sonner |
| DnD | `@dnd-kit/core` (Mi Semana: días) |
| Errores cliente | `@sentry/react` — activo solo si `VITE_SENTRY_DSN` tiene valor; `AppErrorBoundary` en `main.tsx` llama `Sentry.captureException` |

### Variables de entorno (`web/.env`)

- `VITE_INSFORGE_URL`, `VITE_INSFORGE_ANON_KEY` — obligatorias para la app.
- `VITE_ALLOWED_EMAIL_DOMAINS` — opcional (whitelist de dominios de email).
- `VITE_SENTRY_DSN` — opcional; vacío = Sentry no inicializa (sin coste ni ruido).

### Cliente InsForge y sesión

- Cliente singleton; `persistSession` / `autoRefreshToken` según reglas del proyecto.
- `src/lib/insforgeFetchInterceptor.ts`: en respuestas 401 a `/api/database/`, refresh de sesión y un reintento; si falla, cierre de sesión y redirección a login (no mostrar errores técnicos de token al usuario).
- `AuthProvider`: bootstrap con `getCurrentUser` + `asegurarUsuario`; **verificación periódica** cada **4 minutos** con `getCurrentUser()` — si falla o no hay usuario, `clear()` (logout lógico).

## 2. Arquitectura del monorepo

```
agenda-ti_v3/
├── db/                    # schema.sql, seed.sql, migrations/
│   └── migrations/        # Incl. 024: logs inmutables + validación fecha OT
└── web/                   # Esta SPA
    ├── CONTEXT/           # Este documento (snapshots por fecha)
    ├── .env               # No versionar secretos reales
    └── src/
        ├── api/           # InsForge: tablas / RPCs
        ├── components/    # layout, ui, tareas, semana, ot, routing…
        ├── hooks/         # useMiSemanaPage, usePlanificacionPage, etc.
        ├── lib/           # insforge, fechas, permisos, schemas Zod, queryHelpers
        ├── pages/         # Rutas lazy
        ├── providers/
        ├── store/
        ├── types/
        └── styles/        # Sin kanban.css / hoy.css (eliminados en V4)
```

### Patrón de capas

```
Page → useXxxPage (orquestación, queries) → api/*.ts (SDK + parse Zod) → InsForge
```

Las reglas del workspace sugieren features en `src/features/<modulo>/`; el código actual sigue mayormente **pages + hooks + api** (convivencia: nuevas piezas pueden ir a `features/` si se adopta).

## 4. Módulos que **existen** (V4)

### Mi Semana (`/semana`)

- **Solo vista semanal** (Lun–Sáb, 6 columnas): sin toggle “Ahora / Semana”.
- DnD entre columnas de día; contadores de estado; eventos por día (`EventoCard`).
- **Columna del día actual:** bloque **Incidencias** integrado (`ZonaIncidencias`) + acceso a modal de incidencia.
- **Panel lateral de notas** (`PanelNotas`): visible en viewport `xl+`; notas recientes + textarea “nota rápida” + guardar (usa `hoyColumnas` / bitácora como datos, no una página Bitácora).
- Hooks: `useMiSemanaPage`, `useMiSemana`, `useSemanaDnD`, `useSemanaModales`, `useSemanaNavegacion`, `useHoyColumnas` (incidencias/notas del día).
- APIs: `semana.ts`, `tablero.ts` (p. ej. mutaciones que invalidan queries compartidas), `recurrencia.ts`, `hoyColumnas.ts`.

### Objetivos (`/objetivos`)

- CRUD según RBAC (jefe; miembro según políticas en datos).
- APIs: `objetivos.ts`, `objetivosMetricas.ts`.

### Planificación (`/planificacion`) — solo jefe

- Vista de planificación multi-miembro; **solo lectura** en producto (no crear/editar tareas desde ahí).
- API: `planificacion.ts`.
- **Eliminado:** dependencia de `configuracion_semana` / `ConfiguracionSemana` en frontend.

### Métricas (`/metricas`) — solo jefe

- Protegida por `JefeRoute`.

### Órdenes de trabajo (`/ordenes-trabajo`)

- Flujo OT; formulario con validación de **fecha estimada ≥ hoy** (UTC `YYYY-MM-DD` vía `toISOString().slice(0,10)` en cliente).
- API `ordenTrabajo.ts` usa **parsers Zod** (`parseOrdenTrabajo`, `parseTipoTrabajoOT`) — evitar casts ciegos.
- `crearTipoTrabajoOT`: insert solo `{ nombre }` — **sin campo fantasma `created_by`**.

### Autenticación

- `AuthProvider`, `ProtectedRoute`, dominios permitidos en login vía env.

## 6. Reglas de negocio (resumen; detalle en `.cursor/rules`)

- Estados y enums de tareas deben coincidir con **Postgres**; no inventar estados en UI.
- **Tareas atrasadas:** lógica en backend (trigger/cron/RPC); UI refleja `atrasada`.
- **Tarea libre vs planificada:** al asignar a día, tipo y fechas según reglas (sin `tipo` inventado).
- **Imprevistos:** creación con tipo/estado/`es_imprevisto` según reglas.
- **Log obligatorio:** reprogramar, cancelar, bloquear, eliminar → `log_accion` con justificación.
- **Métricas ponderadas:** pesos alta/media/baja definidos en reglas.
- **RBAC:** jefe vs miembro; métricas y planificación acotadas; tablero global: miembro ve todo pero solo edita lo propio (en vistas que apliquen); **RLS en InsForge** es la fuente de verdad, no solo UI.
- **Rol:** desde tabla `public.usuario`, no claims JWT sueltos.
